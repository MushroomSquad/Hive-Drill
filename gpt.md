Это не значит, что он плох для текста. Это значит, что его специализация чуть смещена. Это уже мой вывод на основе того, как они сами позиционируют модель и какие результаты показывают.

Плюс в официальном README у AirLLM среди явно заявленных апдейтов есть Qwen2.5, Llama 3.1, Mixtral и generic AutoModel, но я не нашёл у них явного упоминания ROME как официально проверенной модели. Thought for 1m 33s

Да, и их довольно много. Условно это не одна “ускорялка”, а три этажа турбонаддува:
(1) движок инференса, (2) квантование/кэш, (3) техники декодирования. Самое важное для тебя с RTX 4070 + Cursor-подобным сценарием это не просто “какая модель”, а какой backend + какой формат квантов + есть ли prefix/speculative cache.

Если смотреть на практичные варианты, то вот основные игроки:

ExLlamaV2. Это движок именно под локальный запуск на consumer GPU, а его официальный рекомендуемый сервер это TabbyAPI, который даёт OpenAI-compatible API. Для 4070 это очень вкусный вариант, если ты готов брать модели в GPTQ/EXL2 и хочешь максимум “локальной бодрости” без тяжёлого серверного зоопарка.

llama.cpp. Это швейцарский нож: лёгкий C/C++ сервер, умеет OpenAI-compatible /v1/chat/completions, Flash Attention, prompt cache reuse через cache_prompt, и speculative decoding с draft model или n-gram режимами. Для домашнего сетапа, GGUF-моделей и сценариев “часть в VRAM, часть в RAM” это один из самых удобных путей.

LMDeploy. Недооценённая ракета. В своей документации они заявляют до 1.8x выше request throughput, чем vLLM, за счёт persistent/continuous batching, blocked KV cache, dynamic split&fuse, CUDA kernels, и отдельно пишут, что их 4-bit inference performance до 2.4x выше FP16. Плюс у них есть OpenAI-compatible server. Для NVIDIA-карты это очень сильный кандидат, если хочется именно speed-oriented backend, а не только “чтобы влезло”.

vLLM. Это уже больше про serving throughput, чем про домашнюю магию. Его сильные карты это PagedAttention, continuous batching, prefix caching, chunked prefill, speculative decoding и OpenAI-compatible server. Если у тебя один пользователь в Cursor, он не всегда даст самый сладкий “ощутимый” прирост на 4070, но если запросов несколько, длинные контексты повторяются и нужен серверный режим, это очень сильный вариант.

SGLang. Похож по духу на vLLM, но делает акцент на низкую задержку, высокую пропускную способность, speculative decoding, continuous batching и свою идею RadixAttention для переиспользования KV cache между вызовами. Для IDE-агентов и повторяющихся префиксов это особенно интересно.

TensorRT-LLM. Это уже тяжёлая артиллерия под NVIDIA: в документации у него есть quantization, paged attention / IFB / request scheduling, speculative decoding, chunked context, KV cache system, Torch compile & CUDA graph и свой сервер trtllm-serve с OpenAI-клиентскими примерами. Обычно это вариант для тех, кто хочет выжимать максимум из NVIDIA-стека и готов платить сложностью настройки.

Теперь про техники, которые реально ускоряют, независимо от движка:

Квантование весов. GPTQ, AWQ, EXL2, GGUF-кванты и тому подобные. Это почти всегда первый рычаг, потому что он снижает память и часто даёт прирост скорости. У ExLlamaV2 есть EXL2 с диапазоном от 2 до 8 бит, у LMDeploy есть weight-only и AWQ/GPTQ INT4, а AirLLM сам предлагает 4-bit/8-bit compression и пишет про ускорение до 3x за счёт уменьшения дискового bottleneck при своей layer-wise загрузке.

KV-cache quantization. Это отдельная турбина для длинного контекста. LMDeploy прямо пишет про k/v quantization, а в экосистеме llama.cpp и ExLlama эту тему тоже активно развивают. Особенно полезно, когда у тебя длинные чаты, большие файлы и Cursor постоянно таскает контекст по кругу.

FlashAttention. Это уже почти базовый steroid pack для внимания: официальный проект называет себя fast and memory-efficient exact attention, а llama.cpp умеет включать --flash-attn. Если backend и модель его поддерживают, это одна из самых “чистых” ускорялок без изменения логики работы модели.

Prefix/prompt caching. Для IDE это золото. llama.cpp умеет cache_prompt, где общий префикс не пересчитывается заново, vLLM поддерживает prefix caching, а SGLang продвигает RadixAttention именно как способ автоматического переиспользования KV cache между вызовами. Когда ты гоняешь одни и те же инструкции, кодовую базу и системные промпты, это даёт очень ощутимую выгоду.

Speculative decoding. Суть в том, что маленькая черновая модель или n-gram механизм предлагает токены, а большая быстро подтверждает. TensorRT-LLM, vLLM, SGLang и llama.cpp все в той или иной форме это поддерживают. Когда оно хорошо совпадает с твоей основной моделью, ускорение бывает очень заметным.

Continuous batching / paged attention / chunked prefill. Это серверные ускорялки, которые особенно хороши при нескольких параллельных запросах, стриминге и длинных контекстах. Их официально подчёркивают vLLM, LMDeploy, TensorRT-LLM и SGLang.

Что бы я советовал именно тебе:

Если нужен самый практичный прирост на 4070 для одного пользователя и Cursor, я бы смотрел в таком порядке:
ExLlamaV2 + TabbyAPI, потом llama.cpp, потом LMDeploy. Это мой инженерный вывод из того, как эти проекты сами себя позиционируют: ExLlamaV2 прямо заточен под consumer GPU, llama.cpp очень гибок и умеет OpenAI-compatible API с нужными ускорялками, а LMDeploy выглядит мощно, но уже чуть более “серверно”.

Если приоритет не “максимум tok/s”, а именно впихнуть модель крупнее VRAM, тогда AirLLM остаётся полезным, но его философия другая: он грузит модель послойно, у него bottleneck часто в disk loading, и собственное 4-bit/8-bit compression у них подаётся как способ ускорить именно этот процесс. То есть AirLLM это больше VRAM-спасатель, чем король сырой скорости.

Мой совсем короткий вердикт такой:
для твоего железа самые интересные “ускорялки” это EXL2/GPTQ-кванты + ExLlamaV2/TabbyAPI, либо GGUF + llama.cpp + FlashAttention + prompt cache + speculative decoding. А если хочется уже более серверной математики, смотри LMDeploy или vLLM/SGLang.

Thought for 2m 32s

Если нужен один честный ответ, то для твоей цели лучший по ощущению скорости и “быстрому отклику” стек из того, что мы обсуждали, это:

Harbor + TabbyAPI + ExLlamaV2/V3 + EXL2-кванты.

Это не “абсолютная истина по всем бенчмаркам”, а моя инженерная ставка именно под одну RTX 4070, локальный coding assistant и быстрый интерактив. Причина простая: ExLlamaV2 прямо позиционируется как inference library для modern consumer GPUs, TabbyAPI является официальным backend server для ExLlamaV2/V3, даёт OpenAI-compatible API, сам себя описывает как lightweight and fast, а ещё поддерживает paged attention на NVIDIA Ampere и новее. У Harbor для TabbyAPI есть отдельный backend.

Второе место я бы дал Harbor + llama.cpp + GGUF. Он, скорее всего, будет чуть менее “турборезвым” на чистом consumer-GPU сценарии, чем ExLlama-стек с хорошими EXL2-квантами, но зато это очень живой, реальный и удобный путь: официальный llama.cpp server поддерживает OpenAI-compatible chat/completions/responses, continuous batching, Flash Attention, prompt cache reuse и speculative decoding; в Harbor для него есть backend и даже router mode для нескольких моделей. Если хочется не рекорд, а “поднял и работаю без шаманского бубна”, это очень сильный вариант.

AirLLM я бы не ставил на первое место по скорости. Он очень реальный и полезный, но его суперсила другая: впихнуть большую модель в маленький VRAM. В официальном README AirLLM прямо пишет про компрессию и ускорение, но тут же объясняет, что их bottleneck в основном disk loading, а не “максимально быстрый интерактивный serving”. То есть AirLLM это скорее VRAM-спасатель, чем король мгновенного отклика.

LMDeploy, vLLM и SGLang тоже реальные и воплощаемые, но я бы ставил их ниже для твоего конкретного режима “один пользователь, одна 4070, нужен быстрый local copilot”. LMDeploy официально даёт OpenAI-compatible server и сам заявляет до 1.8x выше request throughput, чем vLLM, плюс 4-bit inference до 2.4x выше FP16. Но это заявление именно про throughput, а не гарантия лучшей single-user latency. vLLM тоже полностью real, у него OpenAI-compatible server и backend в Harbor. SGLang в Harbor тоже есть и прямо описывается как fast serving framework. Я бы смотрел на них, если захочешь потом играться с серверным режимом, параллельными запросами или более тяжёлой serving-архитектурой.

LiteLLM не ускорялка. Это не движок инференса, а gateway/router. Он полезен, если хочешь единый OpenAI-compatible вход, роутинг между локалкой и облаком, fallback и бюджеты. Но сам по себе LiteLLM не сделает локальную модель быстрее. Harbor его поддерживает, а официальные доки LiteLLM называют его self-hosted OpenAI-compatible gateway.

llama.cpp: да, абсолютно реально и очень практично. Есть официальный server и официальный OpenAI-compatible интерфейс.
vLLM: да, реально. Есть официальный OpenAI-compatible server и backend в Harbor.
SGLang: да, реально. Есть backend в Harbor.
AirLLM: сам проект точно real; в Harbor wiki AirLLM backend фигурирует, но в моём просмотре GitHub wiki эта страница рендерилась криво, так что именно Harbor-часть я подтверждаю чуть менее уверенно, чем TabbyAPI/llama.cpp/vLLM.

Мой основной набор моделей для этого стека
1) Основная модель для кода

bartowski/Qwen2.5-Coder-7B-Instruct-exl2 на ветке 6_5

Это мой главный выбор. У самой Qwen2.5-Coder линейки акцент на code generation, code reasoning и code fixing, а у EXL2-кванта 6_5 для этой модели указано примерно 7.2 GB VRAM на 4k, 8.6 GB на 16k и 10.6 GB на 32k. Для 12 GB VRAM это очень приятная зона: модель остаётся бодрой и не душит карту.

Qwen2.5-14B-Instruct лучше подходит под instruction following, длинный текст, таблицы/структурированные данные и JSON, то есть под ТЗ, RFC, планы, описания задач и документацию. Поэтому для 12 GB вариант 4_25 здесь самый рациональный.

3) Если хочешь одну “универсальную” модель вместо двух

bartowski/Qwen2.5-Coder-14B-Instruct-exl2 на 4_25

Это компромиссный вариант “одна модель и для кода, и для текстов”. Но он будет менее шустрым, чем 7B-кодер, и так же ограничен по контексту на 12 GB: у 4_25 для 14B-кодера около 10.1 GB на 4k и 13.0 GB на 16k. Я бы его держал как запасной план, а не как стартовый.

Qwen/Qwen2.5-Coder-14B-Instruct-GGUF в Q4_K_M. У официального GGUF-репо 14B-кодера есть те же кванты, а Q4_K_M для 14B-кодера весит примерно 8.99 GB.

Реальная схема под Harbor + llama.cpp

Harbor позволяет либо указать один GGUF напрямую, либо запустить llama.cpp в router mode. В wiki показано, что router mode включается через очистку llamacpp.model.specifier, после чего можно указывать --models-dir и грузить/выгружать модели через /models/load и /models/unload. Harbor также приводит пример harbor llamacpp args '--max-tokens 1024 -ngl 100', а сам llama.cpp server поддерживает --ctx-size, --flash-attn и --gpu-layers.

Я бы делал так:

# router mode
harbor config set llamacpp.model.specifier ""
harbor llamacpp args "--models-dir /app/data/models --models-max 2 --no-models-autoload -c 8192 -fa on -ngl all"
harbor up llamacpp

Потом кладёшь GGUF-файлы в ./llamacpp/data/models и грузишь нужную модель через API. Harbor показывает для этого /models и /models/load; примеры в wiki идут с портом 33831.

Самый удачный старт

Для твоей RTX 4070 / 49 GB RAM / Ryzen 5600X я бы начал с:

Harbor + TabbyAPI + Qwen2.5-Coder-7B-Instruct-exl2 (ветка 6_5)

Почему именно так: Harbor официально поддерживает TabbyAPI и даёт для него готовые команды скачивания/запуска, TabbyAPI является официальным API-сервером для ExLlamaV2/V3, а квант 6_5 у 7B-кодера укладывается примерно в 7.2 GB VRAM на 4k, 8.6 GB на 16k и 10.6 GB на 32k, что для 12 GB карты очень сладко.

1. Подготовь базу

Harbor в официальной установке просит:

Docker
Docker Compose 2.23.1+
Git
примерно 10 GB+ свободного места для быстрого старта, а в полном гайде ещё фигурирует 20 GB как безопасный запас
для GPU-ускорения на Linux/WSL2 нужен NVIDIA Container Toolkit.
Проверка
docker --version
docker compose version
git --version
nvidia-smi

Если Docker уже есть, Harbor рекомендует ставить CLI либо скриптом, либо через пакетный менеджер:

curl https://av.codes/get-harbor.sh | bash
# или
pipx install llm-harbor
# или
npm install -g @avcodes/harbor

Потом:

harbor --version
harbor doctor

Эти шаги и команды есть в официальном install guide Harbor.

2. Включи GPU для Docker

Если ты на Linux или Windows через WSL2, Harbor отдельно рекомендует NVIDIA Container Toolkit и даёт стандартные команды установки/рестарта Docker. Это нужно, чтобы контейнеры вообще увидели твою 4070.

После установки проверь:

docker run --rm --gpus all nvidia/cuda:12.4.1-base-ubuntu22.04 nvidia-smi

Если внутри контейнера видна 4070, всё хорошо.

3. Поставь Harbor и убедись, что CLI живой
harbor doctor
harbor up

harbor up по умолчанию стартует Ollama + Open WebUI, это просто тест, что стек в принципе живой. Потом мы переключимся на TabbyAPI. Harbor это прямо пишет в user guide.

4. Скачай основную модель для кода

Harbor для TabbyAPI официально предлагает такой путь:

скачать EXL2-модель через harbor hf
указать её как модель сервиса
поднять tabbyapi.
Рекомендованный вариант
harbor hf dl -m bartowski/Qwen2.5-Coder-7B-Instruct-exl2 -s ./hf -b 6_5
harbor tabbyapi model bartowski/Qwen2.5-Coder-7B-Instruct-exl2
harbor up tabbyapi

Harbor в wiki показывает именно такой паттерн команд: harbor hf dl ... -s ./hf -b ..., потом harbor tabbyapi model ..., потом harbor up tabbyapi. У самого сервиса адрес по умолчанию http://localhost:33931.

5. Проверь, что endpoint дышит

TabbyAPI у Harbor заявлен как OpenAI-compatible API.

Минимальная проверка:

curl http://localhost:33931/v1/models

Потом тестовый запрос:

curl http://localhost:33931/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen2.5-Coder-7B-Instruct-exl2",
    "messages": [
      {"role": "system", "content": "You are a concise coding assistant."},
      {"role": "user", "content": "Write a Python function that chunks a list into batches of size n."}
    ],
    "temperature": 0.2,
    "max_tokens": 300,
    "stream": false
  }'

Если сервис стартовал, ты увидишь обычный chat completion JSON.

6. Добавь вторую модель для ТЗ и документации

Для текстов, ТЗ, RFC и доков я бы держал вторую модель:

bartowski/Qwen2.5-14B-Instruct-exl2 ветка 4_25

У неё квант 4_25 занимает примерно 10.1 GB VRAM на 4k и 13.0 GB на 16k. Это уже на грани для 12 GB VRAM, но как “вторая, более тяжёлая модель для текстов” вариант реальный.

Скачать:

harbor hf dl -m bartowski/Qwen2.5-14B-Instruct-exl2 -s ./hf -b 4_25

Когда захочешь переключиться на неё:

harbor tabbyapi model bartowski/Qwen2.5-14B-Instruct-exl2
harbor up tabbyapi
7. Что оставить как основной режим

Я бы оставил такой расклад:

основной daily driver для кода
Qwen2.5-Coder-7B-Instruct-exl2@6_5
режим для ТЗ/доков/архитектуры
Qwen2.5-14B-Instruct-exl2@4_25

Это самый практичный набор под твоё железо из того, что реально поднимается через Harbor и не превращает 4070 в кашляющий самовар.

8. Если захочешь более надёжный запасной путь

Если TabbyAPI вдруг начнёт капризничать, запасной трактор такой:

Harbor + llama.cpp + GGUF

Это тоже полностью реальный путь. У официального llama.cpp server есть OpenAI-compatible chat completions, responses и embeddings, а Harbor поддерживает его как backend и умеет даже router mode для нескольких GGUF-моделей.

Быстрый запасной кодер

bartowski/Qwen2.5-Coder-7B-Instruct-GGUF
Файл Q5_K_M весит 5.44 GB, Q4_K_M весит 4.68 GB.

Запасной текстовый режим

bartowski/Qwen2.5-14B-Instruct-GGUF
Файл Q4_K_M весит 8.99 GB, Q5_K_M весит 10.51 GB.

Запуск через Harbor можно сделать либо прямым URL на GGUF, либо router mode. Harbor официально описывает оба варианта.

9. Как подключить это к Cursor

Тут есть важная заноза.

По словам сотрудника Cursor на их форуме:

custom OpenAI-compatible endpoint поддерживается через Override OpenAI Base URL
но localhost и LAN IP напрямую сейчас не поддерживаются
нужен публично доступный HTTPS endpoint
в ряде случаев Cursor в Agent mode шлёт Responses API-формат, и если твой backend умеет только /v1/chat/completions, запрос может упасть
при проблемах они рекомендуют переключить HTTP Compatibility Mode на HTTP/1.1.
Что это значит practically

Для прямого локального теста используй:

curl
Open WebUI
любые клиенты, которым хватает http://localhost:33931/v1

Для Cursor тебе понадобится:

поднять публичный HTTPS endpoint поверх твоего локального сервера
прокинуть его на .../v1
в Cursor:
открыть Settings → Models
включить Override OpenAI Base URL
указать этот публичный https://.../v1
ввести любой нужный API key, если твой прокси его требует
добавить/выбрать модель
если не взлетело:
включить HTTP/1.1
начать с Ask mode, а не Agent mode.
Важный нюанс

Если хочешь, чтобы Cursor Agent работал более предсказуемо, то запасной вариант через llama.cpp может оказаться удобнее, потому что официальный llama.cpp server уже заявляет поддержку и chat completions, и responses routes.
